Implementation Approach

Cách tổ chức code và refactor từ flow hiện tại sang flow mới

A

Incremental Refactor

Giữ nguyên structure hiện tại, mở rộng dần:

  • Mở rộng DesignEntry → thêm frames: FrameEntry[], mỗi frame có notes + tags
  • Refactor figma-annotator.tsx — thêm view mode state (grid/detail), update sidebar thành tree
  • Tách annotations-panel thành 2 component: DesignTreeSidebar + FrameAnnotationPanel
  • Update handleLoadDesign — thêm smart detection logic

Pros

  • Ít breaking changes, build on existing code
  • Có thể ship từng phần

Cons

  • File figma-annotator.tsx sẽ phình to
  • State management phức tạp hơn trong 1 file
B

Store-based Architecture (Recommended ⭐)

Extract state vào Zustand store riêng, tách UI thành focused components:

  • Tạo use-figma-design-store.ts — quản lý designs, frames, annotations, view mode, selection
  • Data model mới: Design → FrameEntry[] → { notes, tags[] }
  • Tách UI: DesignTreeSidebar, FrameGrid, FrameDetailView, FrameAnnotationPanel, TagPicker
  • figma-annotator.tsx trở thành layout shell, compose các components

Pros

  • Clean separation: state logic tách khỏi UI
  • Components nhỏ, dễ test và maintain
  • Consistent với pattern đã dùng (useTabStore, useStatusBarStore)
  • Dễ extend sau này (persistence, undo/redo...)

Cons

  • Nhiều files mới hơn approach A
  • Cần refactor nhiều hơn upfront
C

Full Rewrite

Viết lại toàn bộ figma-annotator screen từ đầu với architecture mới.

Pros

  • Không bị ràng buộc bởi code cũ
  • Clean architecture từ đầu

Cons

  • Mất hết code đang hoạt động tốt (connection, canvas, tree)
  • Rủi ro regression cao
  • Tốn thời gian nhất
