# Agent Coding — Requirements Summary

Workflow manager for Claude Code sessions. Users create tickets, configure workflow templates, and let AI agents (planner, designer, coder, tester, reviewer) execute tasks autonomously via a DAG-based execution engine.

---

## 1. Tickets

### Ticket Model

| Field | Type | Notes |
|-------|------|-------|
| key | string | Auto-generated: `{PROJECT_SLUG}-{seq}` (e.g. "PROJ-1") |
| title | string(500) | |
| description | text | Markdown |
| type | enum | feature, bug, improvement, chore, spike |
| status | enum | backlog, todo, in_progress, in_review, done, cancelled |
| priority | enum | urgent, high, medium, low, none (default: none) |
| source | enum | ai_brainstorm, figma, manual |
| template_id | FK | Workflow template used to generate steps |
| assignee_id | FK | Optional user assignment |
| budget_usd | decimal | Cost budget |
| auto_execute | bool | Default true — auto-start ready steps |
| auto_approval | bool | Default false — skip post-execution review |
| figma_data | JSON | Figma context if source=figma |

### Ticket Lifecycle

1. Created → status=backlog
2. First step starts → status=in_progress
3. Steps execute via DAG
4. All steps done → status=done (or cancelled)

### Ticket Specs

Versioned specifications attached to tickets.

- **Types:** feature, design, plan, test
- **Versioned:** Each update creates a new record with revision+1 (immutable history)
- **References:** Specs link to each other via typed references (derives_from, implements, verifies, relates_to)
- Specs can be created by users or generated by agent steps

### Ticket Detail UI — 4 Tabs

1. **Overview** — Status, priority, type, source, template, settings (auto-execute, auto-approval, budget), editable description
2. **Specs** — List specs by type, inline edit with rich editor, revision history, spec reference DAG
3. **Tasks** — DAG visualization with live status, step list with edit capability, run/stop/retry/skip actions, cost tracking, code review panel
4. **Activity** — Real-time timeline of all events (step started/completed/failed, spec saved, workflow completed)

### Ticket APIs

```
POST   /projects/:id/tickets              — Create with template
GET    /projects/:id/tickets              — List (filter: status, type, priority, assignee)
GET    /tickets/:id                       — Detail with steps and DAG
PATCH  /tickets/:id                       — Update fields
DELETE /tickets/:id                       — Delete with cascade
POST   /tickets/:id/start                 — Start workflow execution
```

## 2. Workflow Templates

Reusable DAG definitions that generate workflow steps when a ticket is created.

### Template Structure

```json
{
  "steps": [
    {
      "id": "plan",
      "agent": "planner",
      "depends_on": [],
      "description": "Create implementation plan",
      "condition": null,
      "expandable": false
    },
    {
      "id": "design",
      "agent": "designer",
      "depends_on": ["plan"],
      "condition": "has_ui",
      "expandable": false
    },
    {
      "id": "implement",
      "agent": "coder",
      "depends_on": ["plan", "design"],
      "expandable": true
    },
    {
      "id": "test",
      "agent": "tester",
      "depends_on": ["implement"],
      "expandable": true
    }
  ]
}
```

- **condition** — Step only created if condition is met (e.g. `has_ui`, `multi_repo`)
- **expandable** — Planner can split this into multiple sequential sub-tasks

### Default Templates (seeded per project)

| Template | Steps |
|----------|-------|
| Full Feature | plan → design (if has_ui) → implement (expandable) → test (expandable) |
| Bug Fix | plan → fix → test |
| Refactor | plan → test_before → refactor → test_after |

### Validation

- No cycles allowed in DAG
- All `depends_on` references must exist in same template
- All agent references must exist in project

### Template APIs

```
POST   /projects/:id/templates            — Create with DAG validation
GET    /projects/:id/templates            — List all + global
PATCH  /templates/:id                     — Update with re-validation
DELETE /templates/:id                     — Delete (if no active tickets reference it)
```

## 3. Workflow Execution Engine

### Workflow Steps

Steps are instantiated from template when ticket is created.

| Field | Type | Notes |
|-------|------|-------|
| template_step_id | string | References template step config ID |
| name | string | |
| description | text | |
| agent_config_id | FK | Agent that executes this step |
| status | enum | pending, ready, awaiting_approval, running, review, changes_requested, completed, failed, skipped |
| order | int | Execution order |
| requires_approval | bool? | Step-level pre-approval override |
| auto_approval | bool? | Step-level post-approval override |
| user_prompt_override | text? | Custom prompt for agent |
| brainstorm_output | JSON? | |
| step_breakdown | JSON? | |
| retry_count | int | Default 0 |
| max_retries | int | Default 2 |
| parent_step_id | FK? | For auto-created fix tasks (escalation) |
| repo_ids | JSON? | Target repositories |

### DAG Advancement — `tick(ticket_id)`

After any step completes:
1. Examine all pending steps
2. For each pending step: check if ALL dependencies are satisfied (completed)
3. If satisfied: transition pending → ready, schedule execution
4. Update ticket status accordingly (first step → in_progress, all done → done)

Parallel execution is supported — multiple steps can be ready simultaneously.

### Pre-Execution Approval (3-tier resolution)

1. `step.requires_approval` (if set, overrides all)
2. Template step config `requires_approval`
3. `agent_config.default_requires_approval`

If approval required: step goes to `awaiting_approval` instead of auto-starting.

### Post-Execution Approval (2-tier resolution)

1. `step.auto_approval` (if set, overrides ticket)
2. `ticket.auto_approval` (default false → requires review)

If review required: step goes to `review` status after completion. User can approve or request changes.

### Test Failure Escalation Chain

```
Tester runs tests
├── All pass → completed
└── Some fail
    └── Tier 1 (retry_count=0): Retry tester with error context
        ├── Pass → completed
        └── Still fail
            └── Tier 2 (retry_count=1): Auto-create fix task → Coder agent → Re-run tester
                ├── Pass → completed
                └── Tier 3 (retry_count>=2): status=failed, notify user
```

### Code Review Flow

1. Step reaches `review` status
2. ReviewPanel shows diff of changes + test results
3. User comments on specific lines
4. Approve → step completed, DAG advances
5. Request changes → agent regenerates with feedback

### Execution APIs

```
POST   /tickets/:id/run                           — Start workflow (schedule all ready steps)
POST   /tickets/:id/stop                          — Cancel all running sessions
POST   /tickets/:id/steps/:step_id/run             — Manually trigger ready step
POST   /tickets/:id/steps/:step_id/retry           — Retry failed step
POST   /tickets/:id/steps/:step_id/skip            — Skip step, unblock downstream
POST   /tickets/:id/steps/:step_id/approve         — Approve awaiting_approval step
POST   /tickets/:id/steps/:step_id/approve-review  — Approve completed review
POST   /tickets/:id/steps/:step_id/request-changes — Request changes with comments
PATCH  /tickets/:id/steps/:step_id/prompt          — Update user prompt override
```

## 4. Sessions

Sessions represent individual Claude Code agent executions for a workflow step.

### Session Model

| Field | Type | Notes |
|-------|------|-------|
| step_id | FK | Parent workflow step |
| status | enum | running, completed, failed, cancelled |
| git_branch | string | Format: `{ticket-key}/{step-name}` |
| git_commit_sha | string(40) | |
| worktree_path | string | Isolated git worktree path |
| cli_session_id | string | Claude SDK session ID |
| cost_usd | decimal(10,4) | |
| tokens_used | int | |
| started_at | datetime | |
| finished_at | datetime? | |
| exit_code | int? | |
| error_message | text? | |

### Session Messages

Each message in a session is saved:
- **role:** user, assistant, tool
- **content:** text content
- **tool_use:** JSON of tool call details
- **timestamp**

### Git Worktree Isolation

- Each session creates an isolated git worktree at a unique path
- Worktree branch created from merge of all dependency step branches
- Coder tasks sharing a ticket use the same worktree (sequential)
- Designer/Tester get separate worktrees per step

### Claude Code SDK Integration

- Sessions execute via `ClaudeSDKClient` with `ClaudeAgentOptions`
- Agent config provides: system_prompt, claude_model, tools_list, mcp_servers
- Sessions receive context from completed dependency steps (specs)
- Max turns configurable per agent (default 25)

### Streaming & Background Execution

- Sessions enqueued as `run_claude_agent` tasks to ARQ (Redis-backed worker)
- Worker publishes streaming messages to Redis PubSub channel `session:{session_id}`
- FastAPI relays PubSub messages to WebSocket clients
- Frontend receives: assistant messages, tool use events, cost updates

### Session APIs

```
GET    /steps/:step_id/sessions     — List sessions for a step
GET    /sessions/:session_id        — Get session detail with messages
POST   /sessions/:session_id/cancel — Cancel running session
```

### Session Viewer UI

- Displays multiple sessions per step with messages
- Message types: text, tool_call, subagent, todo_list, skill, error
- Collapsible message summaries
- Inspector panel for detailed tool call input/output
- Status badges and token counting

## 5. Brainstorming

AI-guided requirements gathering that produces feature specs and creates tickets.

### Two Entry Paths

**Path A: AI-Driven**
1. User clicks "Create Ticket" → "Start with AI"
2. Opens chat UI with brainstorm agent
3. Agent asks 5-8 structured questions (quiz format), one at a time
4. User selects options or provides free-text
5. Agent generates summary (structured markdown)
6. User reviews/edits in rich editor
7. "Create Ticket" saves feature spec and creates ticket

**Path B: Figma-Driven**
1. User clicks "Create Ticket" → "Start from Figma"
2. Opens Figma viewer — user selects nodes, writes annotations
3. Figma context injected into brainstorm agent
4. Same chat flow continues with design context
5. Result: Feature spec with Design References section

### Quiz Message Format

```typescript
type QuizData = {
  question: string
  context: string          // why this question matters
  options: {
    id: string
    label: string
    description: string
    recommended: boolean
    recommendation_reason: string | null
  }[]
  allow_multiple: boolean
  allow_custom: boolean
}
```

### Message Types

| Type | Description |
|------|-------------|
| text | Plain text response |
| quiz | Structured question with selectable options |
| summary | Final feature spec output |
| figma_context | Design context from Figma |

### Summary Output Format

Generated spec includes sections:
- Problem
- Solution
- Decisions (from quiz answers)
- Requirements
- Design References (if Figma source)
- Acceptance Criteria
- Technical Notes

### Review Phase

After quiz completion:
- Summary displayed in rich text editor
- User can add inline comments on sections
- Batch update sends all comments to agent for revision
- Final "Create Ticket" converts summary to ticket + feature spec

### Brainstorm APIs

```
POST   /projects/:id/brainstorm/start                     — Start session (source: ai/figma, optional figma_data)
POST   /projects/:id/brainstorm/:session_id/message        — Send message / quiz response
POST   /projects/:id/brainstorm/:session_id/complete        — Generate final summary
POST   /projects/:id/brainstorm/:session_id/batch-update    — Update summary with comments
POST   /projects/:id/brainstorm/:session_id/create-ticket   — Create ticket from brainstorm (title, type, priority, template_id)
```

### WebSocket

- Channel: `brainstorm:session`
- Events: `brainstorm_message`, `brainstorm_quiz`, `brainstorm_summary`

## 6. Figma Integration

### Architecture

- Figma plugin connects via WebSocket to plugin server (port 3055)
- Frontend connects directly to plugin server (independent of backend WebSocket)
- UUID-based request/response routing with 15-second timeout

### Figma Viewer UI (3-panel layout)

1. **Left:** Node tree — hierarchical view of all Figma nodes
2. **Center:** Canvas — visual display with node highlighting
3. **Right:** Annotation panel — user writes descriptions for selected nodes

### Plugin Commands

| Command | Description |
|---------|-------------|
| get_selection | Get currently selected frames |
| get_node_info | Get detailed node information |
| export_node_as_image | Export design as image |
| set_selections | Select nodes in Figma |

### Figma Node Data

```typescript
interface FigmaNode {
  id: string
  name: string
  type: string
  absoluteBoundingBox: { x, y, width, height } | null
  fills?: Array<{ type, color }>
  style?: { fontFamily, fontSize, fontWeight, ... }
  characters?: string
  cornerRadius?: number
  children?: FigmaNode[]
}
```

### Figma → Brainstorm Flow

1. User enters Channel ID from Figma plugin
2. Connects and loads selected frame/component
3. User annotates nodes with descriptions
4. Annotations + design exported as markdown via `generateFigmaMarkdown()`
5. Passed to brainstorm session as `figma_data` parameter
6. Saved as brainstorm message with type=`figma_context`
7. Ticket created with `source=figma` and `figma_data` JSON field

### Figma Context Structure

```json
{
  "file_name": "App Design",
  "page_name": "Login",
  "node_names": ["LoginForm", "SignUpButton"],
  "figma_url": "https://figma.com/...",
  "description": "Overall design description"
}
```
